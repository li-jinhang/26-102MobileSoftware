# 智协库员工协作一期开发规划

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档版本 | V1.0 |
| 编写日期 | 2026-07-16 |
| 迭代名称 | 员工协作一期：组织关系与人员目录 |
| 目标系统 | 智协库 HarmonyOS 移动应用 + NestJS 后端 |
| 本期目标 | 让员工可查看直属上下级和所在团队，让主管可查看直接下属，并为二期短期协作建立可靠的人员选择基础。 |

## 2. 背景与现状

智协库目前已具备知识库、流程、通知、考勤和内部邮件等能力，但员工信息仍是平面数据：用户仅保存部门文本和系统角色，没有部门实体、直属上级或下属关系。

当前的 `approver` 是流程审批角色，并不等同于组织上的直属领导；因此不能通过角色推断上下级。示例数据中审批人的 `department` 被写成“部门负责人”，也说明部门、岗位和系统权限尚未分离。

本期先解决“人是谁、属于哪个团队、向谁汇报、管理谁”的基础问题，不实施即时聊天、群组、文件传输或协作议题。二期将在本期组织数据的基础上实现面向短期需求的协作沟通。

## 3. 目标与非目标

### 3.1 本期目标

1. 建立部门树和员工直属汇报关系。
2. 员工可查看本人所属部门、直属上级和同部门成员概览。
3. 主管可查看直属下属及其基本信息。
4. 管理员可维护部门和员工组织关系。
5. 通过统一的人员目录接口，为邮件收件人选择、流程审批人定位和二期协作参与人选择提供数据基础。
6. 保证组织关系与系统角色、流程审批角色彼此独立。

### 3.2 非目标

- 不实现实时聊天、WebSocket、语音/视频通话。
- 不实现跨部门矩阵汇报、多直属上级或复杂项目虚线汇报。
- 不实现完整 HR 人事档案、入转调离审批。
- 不在本期重构现有流程审批引擎；流程仍按现有审批角色规则运行。
- 不新增底部一级导航，避免扩大现有移动端信息架构。

## 4. 业务规则

### 4.1 组织关系

1. 每位在职员工归属一个部门，部门可以有一个父部门，形成树形结构。
2. 每位员工最多拥有一个直属上级；顶级负责人可不设置直属上级。
3. 部门负责人由 `leaderId` 指定，其可以与员工的直属上级相同，也可以不同。
4. `role` 仅用于系统权限控制，`managerId` 仅用于组织汇报关系，两者不得互相推导。
5. 不允许员工将自己设为直属上级；不允许形成 A 管理 B、B 管理 A 的循环关系。
6. 员工离职或停用后不应出现在人员选择器中；历史流程和邮件中的姓名快照保留。

### 4.2 可见范围

| 访问者 | 可查看范围 | 可维护范围 |
|---|---|---|
| 普通员工 | 本人、直属上级、同部门在职成员基本信息 | 无 |
| 主管 | 本人、直属上级、同部门成员、直接下属 | 无 |
| 系统管理员 | 全部部门和在职/停用员工 | 部门、员工部门归属、直属上级、状态 |
| 知识管理员 | 与普通员工一致 | 无 |

基本信息仅包括姓名、岗位、部门、直属上级/直属下属关系和在职状态；不展示密码、权限列表等敏感字段。

## 5. 数据模型设计

### 5.1 部门模型

新增 `Department` 模型：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 主键，例如 `d-rd` |
| `name` | string | 部门名称 |
| `parentId` | string nullable | 上级部门 ID；顶级部门为空 |
| `leaderId` | string nullable | 部门负责人用户 ID |
| `sortOrder` | int | 同级展示顺序 |
| `status` | enum | `active` / `inactive` |

### 5.2 用户模型扩展

现有 `User` 增加以下字段，并保留旧字段以兼容当前页面：

| 字段 | 类型 | 说明 |
|---|---|---|
| `departmentId` | string | 所属部门 ID，替代仅靠部门名称关联 |
| `managerId` | string nullable | 直属上级用户 ID |
| `jobTitle` | string | 岗位名称，例如“产品经理” |
| `employmentStatus` | enum | `active` / `inactive` |

现有 `department: string` 暂时保留为展示冗余字段，接口返回时由部门实体生成。完成页面迁移后，可在后续版本逐步移除该冗余字段。

### 5.3 Prisma 关系建议

在 `server/prisma/schema.prisma` 中补充自关联和部门关系。字段命名可按项目现有风格调整，但必须保留以下关系语义：

```prisma
enum EmploymentStatus {
  active
  inactive
}

model Department {
  id        String       @id
  name      String
  parentId  String?
  leaderId  String?
  sortOrder Int          @default(0)
  status    String       @default("active")
  parent    Department?  @relation("DepartmentTree", fields: [parentId], references: [id])
  children  Department[] @relation("DepartmentTree")
  users     User[]       @relation("UserDepartment")
}

model User {
  // 保留已有字段
  departmentId     String?
  managerId        String?
  jobTitle         String @default("")
  employmentStatus EmploymentStatus @default(active)
  departmentEntity Department? @relation("UserDepartment", fields: [departmentId], references: [id])
  manager          User? @relation("UserManager", fields: [managerId], references: [id])
  directReports    User[] @relation("UserManager")
}
```

说明：首轮开发可继续以 `mock-db.ts` 为数据源验证业务闭环；同时更新 Prisma schema，为后续真实持久化做好迁移准备。不要在本期混合完成数据库落地与全部页面重构。

### 5.4 前端模型

在 `entry/src/main/ets/models/AppModels.ets` 中扩展 `User`，并新增：

```ts
export type EmploymentStatus = 'active' | 'inactive';

export interface Department {
  id: string;
  name: string;
  parentId: string;
  leaderId: string;
  sortOrder: number;
  status: 'active' | 'inactive';
}

export interface EmployeeDirectoryItem {
  id: string;
  name: string;
  departmentId: string;
  departmentName: string;
  jobTitle: string;
  managerId: string;
  managerName: string;
  employmentStatus: EmploymentStatus;
}

export interface OrganizationProfile {
  employee: EmployeeDirectoryItem;
  manager: EmployeeDirectoryItem | null;
  directReports: EmployeeDirectoryItem[];
  departmentMembers: EmployeeDirectoryItem[];
}
```

前端对外使用 `EmployeeDirectoryItem`，避免将 `password`、`permissions` 等登录用户对象直接暴露给人员目录页面。

## 6. 接口规划

接口统一沿用现有鉴权方式：除登录接口外，均携带 `Authorization: Bearer <token>`。响应继续使用 `{ code, message, data }` 结构。

### 6.1 获取我的组织关系

`GET /organization/me`

返回当前员工、直属上级、直属下属和同部门成员概览。个人中心和工作台使用该接口。

### 6.2 获取部门树

`GET /organization/departments/tree`

返回启用部门的树形结构。普通员工接口只返回其有权查看的分支；管理员返回完整树。

### 6.3 获取人员目录

`GET /organization/users?scope=department&keyword=张`

查询参数：

| 参数 | 可选值 | 说明 |
|---|---|---|
| `scope` | `department` / `team` / `all` | 可见范围；`all` 仅管理员可用 |
| `departmentId` | string | 按部门过滤 |
| `keyword` | string | 匹配姓名、账号或岗位 |
| `activeOnly` | boolean | 默认 `true` |

本接口将在二期复用为协作参与人选择器，并可替换邮件页中的“手输收件人 ID”。

### 6.4 管理员维护接口

```text
POST  /organization/departments
PATCH /organization/departments/:id
PATCH /organization/users/:id/organization
```

`PATCH /organization/users/:id/organization` 接收 `departmentId`、`managerId`、`jobTitle`、`employmentStatus`。服务端必须校验目标部门存在、上级在职、员工与上级不相同、汇报链无环。

## 7. 前端页面与交互规划

### 7.1 个人中心

修改 `features/profile/ProfilePage.ets`：

1. 在头像与角色信息下新增“我的组织”区块。
2. 展示所属部门、岗位和直属上级姓名。
3. 有直属下属时显示“管理 N 人”，点击进入团队成员页。
4. 点击直属上级或部门可进入组织详情页。

### 7.2 工作台

修改 `features/dashboard/DashboardPage.ets`：

1. 在现有待办、邮件等摘要区域增加“团队关系”轻量卡片。
2. 员工显示直属上级和部门；主管额外显示直接下属数量。
3. 卡片点击进入组织详情页，不新增底部导航项。

### 7.3 组织详情页

新增 `features/organization/OrganizationPage.ets`，包含两个视图：

1. “我的关系”：上级、本人、直接下属的关系链；普通员工仅展示上级和本人。
2. “部门成员”：按岗位展示当前部门在职成员，支持关键词筛选。

页面只展示组织信息，不在本期提供发消息、转交任务等操作，以保持迭代范围稳定。

### 7.4 路由与状态

在 `pages/Index.ets` 中增加 `organization` 子页面状态和返回逻辑；组织数据通过新增 UseCase 和 Repository 加载。页面不得直接读取 `mock/MockData.ets`。

## 8. 后端实现规划

### 8.1 模块划分

新增 `server/src/modules/organization/`：

```text
organization/
├─ organization.controller.ts
├─ organization.service.ts
├─ organization.repository.ts
├─ organization.dto.ts
└─ organization.mapper.ts
```

Controller 仅负责参数和响应；Service 负责权限、可见范围和循环校验；Repository 屏蔽内存 Mock 与 Prisma 的差异。

### 8.2 Mock 数据阶段

1. 在 `server/src/data/mock-db.ts` 增加部门、岗位和上下级示例数据。
2. 提供 `getMyOrganization`、`getDepartmentTree`、`searchDirectory` 等函数。
3. 对既有四个演示账号补充组织关系，例如：张晓宁直属上级为李清越，李清越属于产品研发部并管理张晓宁。
4. 在 Mock 阶段完成接口、权限和 UI 验证后，再切换 Prisma Repository。

### 8.3 Prisma 迁移阶段

1. 执行 schema migration，生成 `Department` 表与 `User` 新字段。
2. 编写 seed 数据，数据结构与 Mock 示例保持一致。
3. 以 Repository 实现替换内存读写，不改变 Controller、前端接口和页面模型。

## 10. 测试与验收标准

### 10.1 功能验收

1. `employee01` 登录后可看到产品研发部、直属上级李清越，且看不到无关部门成员。
2. `manager01` 登录后可看到自身所在部门、直属上级（如有）和张晓宁等直接下属。
3. 管理员可新增部门、调整员工直属上级和停用员工。
4. 修改员工直属上级后，员工和新主管的组织页重新加载即可反映新关系。
5. 非管理员调用全量目录或维护接口时，服务端返回无权限错误。
6. 设置本人为上级、设置不存在的上级、形成循环汇报关系时，服务端拒绝请求。

### 10.2 回归验收

1. 现有登录、知识库、流程、通知、考勤和邮件页面可正常打开。
2. `approver` 角色仍可处理原有流程待办，不因 `managerId` 缺失或变动失效。
3. 既有邮件收发和通知跳转不依赖组织关系即可正常使用。
4. 个人中心原有角色权限、邮件和消息入口保持可用。

## 11. 风险与控制措施

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 将审批角色当作组织上级 | 上下级信息错误，审批规则被误改 | 始终使用 `managerId` 表达直属关系，流程继续使用 `role` |
| 一期同时上实时聊天 | 工期和状态复杂度显著增加 | 本期仅完成组织关系与目录，沟通留到二期协作议题 |
| 直接删除旧 `department` 文本字段 | 现有页面和接口大面积回归 | 采用兼容期，先增加 `departmentId`，后续再清理冗余字段 |
| 部门树和汇报链数据不一致 | 页面显示矛盾 | 后端维护接口执行部门存在性、人员状态和无环校验 |
| Mock 与 Prisma 数据行为不一致 | 后续切库风险高 | 使用相同 DTO、相同 Repository 契约和同一套 seed 样例 |

## 12. 二期衔接

一期完成后，二期可直接复用 `EmployeeDirectoryItem` 和人员目录接口，实现“协作议题”的参与人选择、直属上级快速邀请、团队内问题求助和流程关联讨论。届时新增的消息、未读和权限均应基于具体参与人，而不是目前通知模块的角色广播方式。
