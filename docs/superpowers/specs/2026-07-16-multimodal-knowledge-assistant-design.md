# 智能知识助手统一文字、图片与流程预填设计

## 1. 目标

将知识库顶部的文字搜索和图片 OCR 入口迁移到工作台的“智能知识助手”，形成统一的文字与图片大模型入口。助手既能回答公司知识问题，也能识别请假、报销、采购和权限申请意图，并允许用户选择手动填写或使用大模型草稿预填表单。

## 2. 范围

本次包含：

- 智能助手支持文字输入和相册图片输入。
- 两种输入都通过后端调用现有大模型配置。
- 返回知识回答、图片识别文字、关联知识和推荐流程。
- 推荐流程提供“打开表单”和“智能预填”两个入口。
- 知识库移除搜索框和两个图片按钮，保留分类浏览与知识列表。
- 保留现有流程提交、审批、安全确认和审计逻辑。

本次不包含：

- 大模型自动提交流程。
- 大模型直接修改数据库或审批状态。
- 向量数据库或独立知识嵌入服务。
- 语音输入、连续多轮 Agent 执行和后台自动任务。

## 3. 核心交互

### 3.1 文字输入

用户在智能助手输入问题或待处理事项，例如：

```text
我明天下午身体不舒服，需要请病假去医院。
```

后端调用大模型，返回回答、关联制度、请假流程推荐和可用的表单草稿。

### 3.2 图片输入

用户从相册选择发票、行程单、制度截图或包含业务描述的图片。应用压缩并编码图片后发送给同一助手接口。大模型直接读取图片并一次完成文字识别、知识回答、流程意图判断和草稿字段提取，不再先调用 OCR 再进行第二次大模型请求。

### 3.3 流程操作

只有模型返回合法的流程模板 ID 时才显示以下操作：

- `打开表单`：打开对应流程，使用现有默认值，用户自行填写。
- `智能预填`：将后端校验后的草稿字段写入表单，缺失字段保持为空并提示用户补充。

两种方式都只进入编辑状态。用户必须检查表单并主动点击“提交流程”，大模型不能自动提交。

## 4. 页面设计

### 4.1 智能知识助手

页面包含：

- 可输入多行文字的输入区域。
- `从相册选择`按钮。
- `发送`按钮。
- 图片预览和识别文字区域，仅图片输入时显示。
- 大模型回答。
- 关联知识卡片，可进入知识详情。
- 推荐流程区域。
- `打开表单`和`智能预填`按钮。
- 最近问答历史。
- 请求中、请求失败、无匹配流程和草稿字段缺失状态。

允许用户只提交文字、只提交图片，或在选择图片后补充一段说明。

### 4.2 知识库

删除：

- 文字搜索输入框。
- `从相册选择`按钮。
- `使用内置演示图片`按钮。

保留：

- 分类按钮。
- 按分类浏览知识。
- 知识列表、收藏状态和知识详情入口。

## 5. 后端接口

新增统一接口：

```http
POST /assistant/analyze
Authorization: Bearer <token>
Content-Type: application/json
```

请求：

```json
{
  "text": "我明天下午需要请病假",
  "imageBase64": "",
  "imageMimeType": ""
}
```

`text` 和 `imageBase64` 至少提供一个。图片输入允许同时提供补充文字。

响应业务数据：

```json
{
  "answer": "根据请假制度，病假需要说明时间和原因。",
  "recognizedText": "",
  "intent": "leave",
  "relatedKnowledgeIds": ["k1"],
  "recommendedWorkflowId": "wf1",
  "confidence": 0.92,
  "formDraft": {
    "title": "病假申请",
    "extra": "病假",
    "date": "2026-07-17 14:00-18:00",
    "amount": "",
    "reason": "身体不适，需要就医",
    "attachment": ""
  },
  "missingFields": []
}
```

固定意图枚举：

```text
knowledge_query
leave
reimbursement
purchase
permission
unknown
```

## 6. 大模型调用与约束

后端抽取通用大模型客户端，OCR 和智能助手共用现有的 `LLM_BASE_URL`、`LLM_API_KEY` 和 `LLM_MODEL`。智能助手使用独立提示词和独立响应解析器。

每次请求向模型提供：

- 当前日期和时区。
- 已发布知识的 ID、标题、摘要、标签和必要内容。
- 允许推荐的流程模板 ID、名称、字段含义和风险信息。
- 固定 JSON Schema 和固定意图枚举。

当前知识量较少，直接提供受控知识上下文即可，不引入向量数据库。后续知识量明显增长时再增加检索增强。

后端必须执行以下校验：

- 去除 Markdown 代码块并解析 JSON。
- 校验字段类型、长度和枚举。
- 只保留数据库中存在且已发布的知识 ID。
- 只接受 `wf1` 至 `wf4` 中真实存在的流程 ID。
- 金额字段必须为非负数字字符串。
- 图片格式限制为 JPEG、PNG，限制编码后大小。
- 非法流程 ID、非法字段和模型额外输出必须丢弃。
- 大模型调用失败时返回受控错误，不生成虚假的成功结果。

## 7. 前端状态与表单映射

扩展助手响应模型，加入输入来源、识别文字、意图、置信度、表单草稿和缺失字段。

流程草稿映射：

| 大模型字段 | 现有流程表单状态 |
| --- | --- |
| `title` | `workflowTitleInput` |
| `extra` | `workflowExtraInput` |
| `date` | `workflowDateInput` |
| `amount` | `workflowAmountInput` |
| `reason` | `workflowReasonInput` |
| `attachment` | `workflowAttachmentInput` |

新增两条明确导航：

- `openWorkflowWithoutDraft(templateId)`：使用现有默认表单状态。
- `openWorkflowWithDraft(templateId, draft)`：先建立对应模板状态，再覆盖模型提供的非空字段。

模型没有提供的字段不使用猜测值覆盖。流程模板原有风险提示、审批链和安全确认保持不变。

## 8. 历史记录

继续使用现有助手历史记录。文字请求保存原问题；图片请求保存识别文字的摘要或“图片查询”。第一版不保存图片二进制，避免扩大数据库和隐私范围。

## 9. 错误处理

- 未输入文字且未选择图片：前端阻止提交。
- 图片无法读取或超过限制：前端提示重新选择。
- 大模型未返回合法 JSON：后端返回标准业务错误。
- 未识别到流程：仍显示知识回答，不显示流程按钮。
- 识别到流程但字段不足：允许智能预填已知字段，同时列出待补充项。
- 网络失败：保留当前输入，允许用户重试。

## 10. 验收标准

- 知识库保留分类浏览，搜索框和两个图片按钮不再出现。
- 智能助手可以通过文字调用真实大模型并展示回答。
- 智能助手可以从相册选择图片并展示识别文字和回答。
- 文字和图片均可推荐合法知识文章。
- 请假、报销、采购和权限场景均能进入正确表单。
- `打开表单`不应用模型草稿。
- `智能预填`只填入模型已确认的字段，用户仍可修改。
- 任意助手操作都不能绕过用户点击直接提交流程。
- 大模型返回非法 JSON、非法知识 ID 或非法流程 ID 时，应用不崩溃且不执行错误跳转。
- 前端主 HAP、ohosTest HAP和后端 TypeScript 构建均成功。
