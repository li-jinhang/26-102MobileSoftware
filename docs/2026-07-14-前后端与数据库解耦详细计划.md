# 智协库前端、后端与数据库解耦详细计划

## 1. 目标

当前项目已经完成 HarmonyOS 端原型与本地 Mock 驱动的业务闭环。下一阶段的核心目标，是把“页面展示”“业务逻辑”“数据存取”三层彻底拆开，使项目从课程演示型原型，演进为可继续扩展、可接入真实服务、可替换数据库实现的结构。

本次解耦要达成 4 个结果：

1. 前端不再直接依赖 `MockData.ets`。
2. 前端不再自己拼装复杂业务规则和流程状态。
3. 后端通过明确的 Service / Repository 边界管理业务与数据访问。
4. 数据库更换、表结构调整、接口地址变化时，不需要大面积修改前端页面。

## 2. 当前现状与主要耦合点

结合当前仓库，主要耦合点如下：

### 2.1 页面入口承担了过多职责

`[entry/src/main/ets/pages/Index.ets](file:///d:/_Projects/_assignment/Mobile_software_development/entry/src/main/ets/pages/Index.ets)` 目前同时承担了：

- 登录态管理
- 页面路由切换
- 工作台、知识库、流程、消息、个人中心的状态管理
- 流程创建、审批、二次确认等业务规则
- 通知生成与审计日志写入
- 与 Mock 数据的直接绑定

这意味着：

- UI 修改容易误伤业务逻辑
- 接口接入时，`Index.ets` 会继续膨胀
- 很难单独测试流程规则和数据读写

### 2.2 前端直接依赖 Mock 数据源

`[entry/src/main/ets/mock/MockData.ets](file:///d:/_Projects/_assignment/Mobile_software_development/entry/src/main/ets/mock/MockData.ets)` 当前同时承担了：

- 模拟数据库
- 认证逻辑
- 工作流实例构建
- 智能助手回复规则
- 初始数据导出

这会导致：

- “数据源”与“业务规则”混在一起
- 后续换成真实 API 时，前端要大量改动 import 和调用方式
- 很难区分哪些逻辑应该迁到后端

### 2.3 服务层还偏工具函数，不是完整业务边界

当前已有：

- `[entry/src/main/ets/common/services/KnowledgeService.ets](file:///d:/_Projects/_assignment/Mobile_software_development/entry/src/main/ets/common/services/KnowledgeService.ets)`
- `[entry/src/main/ets/common/services/AssistantService.ets](file:///d:/_Projects/_assignment/Mobile_software_development/entry/src/main/ets/common/services/AssistantService.ets)`
- `[entry/src/main/ets/common/services/MessageService.ets](file:///d:/_Projects/_assignment/Mobile_software_development/entry/src/main/ets/common/services/MessageService.ets)`

但这些更多是前端纯函数工具，还没有形成：

- `API Service`：负责请求远端接口
- `Domain Service`：负责业务规则编排
- `Repository`：负责屏蔽数据源差异

### 2.4 数据模型直接面向页面展示

`[entry/src/main/ets/models/AppModels.ets](file:///d:/_Projects/_assignment/Mobile_software_development/entry/src/main/ets/models/AppModels.ets)` 当前模型基本是前端展示模型。后续接入数据库时，会出现：

- 数据库存储字段
- 接口返回字段
- 前端页面展示字段

三者不完全一致的问题。如果没有 DTO / Entity / ViewModel 分层，后面会越来越难维护。

## 3. 解耦后的目标架构

建议采用四层结构：

1. 表现层 `Presentation`
2. 应用层 `Application`
3. 领域层 `Domain`
4. 基础设施层 `Infrastructure`

### 3.1 前端职责

前端只负责：

- 页面展示
- 用户交互
- 调用前端应用服务
- 渲染后端返回的数据

前端不再负责：

- 认证判定规则
- 审批流转规则
- 风险阈值判定
- 通知生成规则
- 审计日志落库逻辑

### 3.2 后端职责

后端负责：

- 用户认证
- 知识库查询
- 工作流发起、审批、驳回、二次确认
- 消息通知生成
- 审计日志记录
- 权限控制

### 3.3 数据库职责

数据库只负责持久化：

- 用户
- 知识分类与知识文章
- 流程模板
- 流程实例
- 审批记录
- 通知
- 收藏、纠错、问答历史
- 审计日志

数据库不承载页面逻辑，不与前端直接交互。

## 4. 推荐的目录重构方案

## 4.1 前端目录

建议在 `entry/src/main/ets/` 下逐步调整为：

```text
entry/src/main/ets/
├─ app/
│  ├─ api/                # 只负责 HTTP 请求
│  ├─ repositories/       # 屏蔽 mock / remote 差异
│  ├─ usecases/           # 前端应用层用例
│  └─ mappers/            # DTO -> ViewModel
├─ common/
│  ├─ components/
│  ├─ constants/
│  ├─ utils/
│  └─ services/           # 保留纯前端工具服务
├─ domain/
│  ├─ models/             # 领域模型
│  └─ contracts/          # Repository 接口约定
├─ features/
│  ├─ auth/
│  ├─ dashboard/
│  ├─ knowledge/
│  ├─ workflow/
│  ├─ messages/
│  ├─ profile/
│  └─ assistant/
├─ mock/
│  └─ adapters/           # 仅开发阶段启用
└─ pages/
```

### 4.2 后端目录

如果后端准备单独新建服务，建议结构如下：

```text
server/
├─ src/
│  ├─ modules/
│  │  ├─ auth/
│  │  ├─ users/
│  │  ├─ knowledge/
│  │  ├─ workflow/
│  │  ├─ notifications/
│  │  ├─ audit/
│  │  └─ assistant/
│  ├─ common/
│  │  ├─ middleware/
│  │  ├─ errors/
│  │  └─ utils/
│  ├─ database/
│  │  ├─ entities/
│  │  ├─ repositories/
│  │  └─ migrations/
│  └─ app.ts
└─ docs/
```

后端技术可选：

- `Node.js + NestJS + Prisma + SQLite`
- 或 `Node.js + Express + Prisma + SQLite`

如果目标是课程设计与演示，优先推荐 `NestJS + Prisma + SQLite`，原因是分层清晰、文档好，而且不需要额外安装独立数据库服务，最适合本地开发、演示和阶段性验收。

## 5. 需要先定义的边界

在真正改代码前，先把以下边界定死。

### 5.1 前端只依赖 Repository 接口，不直接依赖数据源

例如：

- `AuthRepository`
- `KnowledgeRepository`
- `WorkflowRepository`
- `NotificationRepository`
- `AssistantRepository`

前端页面调用链调整为：

`Page -> UseCase -> Repository -> MockAdapter / HttpApi`

而不是现在的：

`Page -> MockData / Page -> 业务规则`

### 5.2 后端只依赖 Repository，不直接把 SQL 写进 Controller

后端调用链建议为：

`Controller -> ApplicationService -> DomainService -> Repository -> Database`

### 5.3 数据传输对象单独定义

至少拆成三类模型：

- `Entity`：数据库实体
- `DTO`：接口输入输出
- `ViewModel`：前端页面展示对象

比如 `WorkflowInstance` 后续至少会拆为：

- `WorkflowInstanceEntity`
- `WorkflowInstanceResponseDto`
- `WorkflowCardViewModel`

## 6. 数据库初步设计

建议首版最少落这些表：

1. `users`
2. `roles`
3. `knowledge_categories`
4. `knowledge_articles`
5. `workflow_templates`
6. `workflow_instances`
7. `approval_records`
8. `notifications`
9. `knowledge_favorites`
10. `knowledge_corrections`
11. `assistant_histories`
12. `audit_logs`

### 6.1 关键关系

- 一个用户可以发起多个流程实例
- 一个流程实例对应多个审批记录
- 一个知识文章属于一个分类
- 一个知识文章可关联多个流程模板
- 一个用户可收藏多个知识文章
- 一个通知可面向一个用户，也可面向一个角色

### 6.2 当前业务规则哪些必须进数据库，哪些不必

建议进入数据库：

- 用户、角色、权限
- 知识文章与分类
- 流程模板
- 流程实例与审批记录
- 通知与审计日志
- 收藏、纠错、问答历史

建议先不进数据库、保留配置文件或常量：

- 一些纯展示文案
- 前端固定颜色、图标、Tab 文案
- 开发环境接口地址

## 7. 接口拆分规划

建议先按业务模块出接口，不要一开始做成“大而全”的单接口。

### 7.1 认证模块

- `POST /auth/login`
- `GET /auth/me`
- `POST /auth/logout`

### 7.2 知识库模块

- `GET /knowledge/categories`
- `GET /knowledge/articles`
- `GET /knowledge/articles/:id`
- `POST /knowledge/articles/:id/favorite`
- `DELETE /knowledge/articles/:id/favorite`
- `POST /knowledge/articles/:id/corrections`

### 7.3 流程模块

- `GET /workflow/templates`
- `GET /workflow/instances/my`
- `GET /workflow/instances/todo`
- `GET /workflow/instances/:id`
- `POST /workflow/instances`
- `POST /workflow/instances/:id/approve`
- `POST /workflow/instances/:id/reject`
- `POST /workflow/instances/:id/security-confirm`

### 7.4 消息模块

- `GET /notifications`
- `POST /notifications/:id/read`

### 7.5 助手模块

课程设计阶段建议先做“后端规则型助手”，接口如下：

- `POST /assistant/ask`

这样能先把规则从前端移走，后续再替换为真实 LLM 服务。

## 8. 分阶段实施计划

建议分 6 个阶段推进，而不是一次性大改。

### 阶段一：建立解耦骨架

目标：不改业务功能，只改依赖方向。

任务：

1. 新建前端 `repositories`、`usecases`、`api`、`mappers` 目录。
2. 抽离 `AuthRepository`、`KnowledgeRepository`、`WorkflowRepository`、`NotificationRepository` 接口。
3. 新建 `MockRepository` 实现，内部仍调用当前 `MockData.ets`。
4. 页面层改为只调用 `UseCase`，不再直接 import `MockData.ets`。
5. 把 `Index.ets` 中与流程相关的业务方法继续拆到 `workflow/usecases`。

阶段验收：

- 页面功能不变
- `Index.ets` 不再直接 import 大量 Mock 数据方法
- 页面可以在不改 UI 的情况下切换数据源

### 阶段二：前端内部解耦完成

目标：把前端从“大页面管一切”改成“页面只做展示”。

任务：

1. 将登录、知识、流程、消息、个人中心分别建立独立 `UseCase`。
2. 把审批规则、风险判定、通知写入逻辑从 `Index.ets` 抽离。
3. 将 `AppModels.ets` 拆成：
   - domain models
   - response dto
   - view models
4. 为关键 UseCase 补单元测试。

阶段验收：

- `Index.ets` 只负责页面装配、状态绑定、导航
- 业务规则不再堆在 UI 文件中

### 阶段三：后端项目初始化

目标：后端先跑起来，但先只实现最小闭环。

任务：

1. 新建 `server/` 项目。
2. 初始化 `SQLite` 数据库连接、ORM、环境变量。
3. 建立用户、知识、流程、消息、审计几个模块。
4. 先完成登录、知识列表、流程模板三个只读接口。
5. 配置本地 Swagger 或接口文档。

阶段验收：

- 本地后端可以启动
- 前端可以通过 Postman 或浏览器拿到基础接口数据

### 阶段四：数据库建模与迁移

目标：把 Mock 里的核心数据迁到数据库。

任务：

1. 设计首版表结构与外键关系。
2. 编写 migration。
3. 将当前 `MockData.ets` 里的用户、文章、模板、流程实例整理为 seed 数据。
4. 完成数据库初始化脚本。

阶段验收：

- 数据库可一键初始化
- 后端不再依赖硬编码 mock 常量作为主数据源

### 阶段五：前端接入真实后端

目标：把前端数据源从 MockRepository 切到 HttpRepository。

任务：

1. 实现前端 `api/*` 请求类。
2. 实现 `HttpAuthRepository`、`HttpKnowledgeRepository`、`HttpWorkflowRepository` 等。
3. 通过环境配置切换 `mock` / `remote`。
4. 完成登录、知识查询、流程发起、消息已读闭环。

阶段验收：

- 至少一条完整主链路走真实后端
- 页面代码无需因为切源而大改

### 阶段六：业务规则后移与彻底收口

目标：把真正的业务判断全部收口到后端。

任务：

1. 把金额阈值、审批节点流转、二次确认逻辑移到后端。
2. 把通知生成和审计日志写入移到后端。
3. 前端改为只展示后端返回的状态和提示。
4. 清理前端中残留的 mock 业务规则。

阶段验收：

- 前端不再决定流程状态
- 后端成为唯一业务事实来源

## 9. 推荐的迁移顺序

为了降低风险，推荐按下面顺序改：

1. 先抽前端 Repository 接口
2. 再抽前端 UseCase
3. 再拆 `Index.ets`
4. 再建后端骨架
5. 再建数据库和 seed
6. 最后切换真实接口

不要先直接删 `MockData.ets`，否则会同时失去：

- 开发阶段可运行性
- 对照数据
- 回归验证基线

更稳妥的做法是先把它降级成 `MockAdapter`。

## 10. 风险点与规避策略

### 10.1 风险：一次性重构过大，页面全部失稳

规避：

- 每阶段只动一层
- 保留 MockAdapter 作为兜底
- 每完成一个模块就做回归

### 10.2 风险：前端和后端字段不一致

规避：

- 提前定义 DTO
- 加 mapper 层，不在页面直接消费原始接口

### 10.3 风险：流程状态机迁移时出现行为不一致

规避：

- 先把现有流程规则写成后端测试用例
- 用当前 `MockData.ets` 作为规则参考样本

### 10.4 风险：课程项目时间有限

规避：

- 优先完成“登录 + 知识库 + 流程发起 + 消息”的最小闭环
- 助手先保留规则型问答，不急着接大模型

## 11. 验收标准

当以下条件成立，可以认为解耦阶段基本完成：

1. 前端页面不再直接 import `MockData.ets`
2. `Index.ets` 不再承担核心业务规则
3. 前端通过 Repository 切换 mock / remote
4. 后端拥有独立的认证、知识、流程、消息模块
5. 数据库存储用户、知识、流程、通知、审计等核心数据
6. 流程状态流转由后端决定，前端只做展示
7. 换数据库或改接口字段时，不需要大面积改页面组件

## 12. 建议的下一步落地动作

建议你们下一步直接按下面的顺序开始实施：

1. 先做前端内部解耦，不接后端。
2. 第一批只抽 4 个 Repository：认证、知识、流程、消息。
3. 把 `MockData.ets` 改造成 `MockRepository` 的底层实现。
4. 把 `Index.ets` 的流程创建、审批、通知、日志逻辑逐步迁出。
5. 然后我再继续帮你补“后端项目初始化方案 + 数据库表结构设计 + API 文档草案”。

## 13. 结合当前仓库的判断

基于你当前仓库，我建议本轮不要直接进入“前后端联调”，而是先完成“前端内部解耦”。因为现在最明显的问题不是接口缺失，而是：

- 页面入口文件过重
- Mock 与业务规则混合
- 前端暂时没有稳定的数据访问边界

把这一步先做好，后面无论你先接 `SQLite`，还是后续再切到 `MySQL`、`PostgreSQL` 或其他数据库方案，整体成本都会小很多。
